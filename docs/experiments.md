# 실험 설계 및 진행 상황

## 실험 프레임

앱·부하·설정을 전부 동일하게 맞추고, 딱 하나만 다르게 둡니다.

| 항목 | 온프레미스 | EKS |
|---|---|---|
| 파드 자동확장(HPA) | ✅ 동일 | ✅ 동일 |
| **노드 자동확장** ★핵심 | ❌ 없음(고정) | ✅ Karpenter |
| DB 자동복구(선택) | ❌ 수동 | ✅ RDS Multi-AZ |

핵심 독립변수는 "노드 자동확장"입니다. 온프레미스에서 Pending이 쌓이는 것은 실패가 아니라 측정 대상입니다.

## 공통 조건 (양쪽 동일)

| 항목 | 값 |
|---|---|
| 이미지 | GHCR 동일 SHA |
| 부하 도구 | k6(동일 스크립트) — [`../k6/README.md`](../k6/README.md) |
| 리소스 | request 100m/limit 500m, HPA target 70%(min2/max10) |
| 데이터 | 상품 20\~456종, 계정 test1\~2000(`Test1234!`) |
| 실험 전 | `load-test-prep.sql`로 DB 리셋(재고 사전 세팅으로 실험 오염 방지) |

부하 방식은 하이브리드입니다: request를 작게(100m) 둬서 HPA가 민감하게 발동하게 하고, k6 RPS를 크게 줘서 실제 CPU를 올립니다.

## 시나리오 1 — 안정 (베이스라인)

200 RPS를 10\~20분 유지. 목적은 "평상시엔 온프레도 충분하다"는 신뢰 형성입니다. 측정: TPS 안정 / Error율 0% / Pending 0 / 워커 CPU 30\~50%.

## 시나리오 2 — 스파이크 (순간 급증 + 유지) ★핵심

```
0~5분:   200 RPS (안정)
5분:     1500 RPS로 급증 (30초~1분에 걸쳐)
5~15분:  1500 RPS 계속 유지  ← 여기가 진짜 실험(최소 5분)
```

**설계 포인트 — "유지"가 진짜 실험입니다.** 급증을 짧게(10초) 끝내면 시스템이 반응하기 전에 끝나 아무것도 안 보입니다. HPA가 파드를 늘리는 데 시간이 걸리고, Karpenter 노드 생성에는 60~90초가 걸리기 때문에, 급증은 트리거일 뿐 차이는 유지 구간에서 드러납니다.

기대되는 패턴:
- 급증 직후: 양쪽 다 잠깐 에러(파드/노드가 아직 못 따라옴)
- 유지 구간: 온프레는 노드를 못 늘려 Pending이 계속 쌓이고 에러가 지속, EKS는 Karpenter가 노드를 추가(60~90초)해 회복

측정 포인트: 급증 순간의 에러 피크, 유지 구간의 (온프레)에러 지속시간·최대 Pending / (EKS)회복 시간·Node Count 증가.

## 시나리오 3 — 노드 장애 복구 (MTTR)

500 RPS 유지 중 5분 후 worker1을 강제 종료합니다. worker1엔 ingress·gateway·product·inventory가 몰려 있어, 죽으면 모든 파드가 worker2 하나로 재스케줄되며 자원 부족(Pending)이 발생합니다.

- 온프레: worker2 하나로 다 못 들어가 Pending → 수동 복구(수 분)
- EKS: Karpenter가 새 노드를 추가해 파드 분산 → 자동 복구(30~60초)

측정: MTTR(장애 시작→정상 복구), 장애 중 결제 실패 건수(= 비즈니스 손실 환산).

> liveness/readiness 분리가 필수입니다 — `/livez`(DB 안 봄)=liveness, `/health`(DB 봄)=readiness. 안 그러면 DB나 의존 서비스가 죽을 때 전 파드가 재시작 루프에 빠져 측정이 망가집니다.

## 시나리오 4 (선택) — DB 페일오버

핵심은 노드 탄력성(시나리오 2·3)이라 DB 페일오버는 선택 항목이며 실행 여부는 미정입니다. Primary DB를 강제 종료하고, 온프레는 Replica 수동 승격, EKS는 RDS Multi-AZ 자동 복구를 비교합니다. 앱은 `connectionTimeoutMillis: 5000` + `keepAlive`로 죽은 Primary를 빠르게 포기하도록 대응해뒀습니다([`../app/README.md`](../app/README.md) 참고).

## 진행 순서

```
1. 리허설 — 시나리오 2를 작게 1회 → request/RPS·급증속도 다이얼 확정
2. load-test-prep.sql 실행 + 파드 정상 확인
3. 시나리오 1(안정) → Grafana 기록
4. DB 리셋 후 시나리오 2(스파이크) → 급증+유지, 기록 + 스냅샷
5. DB 리셋 후 시나리오 3(노드 장애 → MTTR)
6. 수치 정리 → 결과 리포트
```

지금까지 확인된 것:
- HPA가 부하에 반응해 파드가 늘어나는 것을 확인(최대 약 60개까지 증가) — 초기에는 Pending·스케줄 실패가 함께 발생한 상태로, "부하가 실제로 들어가고 파드가 늘어나는지"를 확인한 수준이었습니다.
- 워커 노드 CPU가 최대 80~90%까지 올라가는 것을 Grafana에서 실시간 관찰([`../onprem/images/hpa-under-load.png`](../onprem/images/hpa-under-load.png)).
- k6로 1800 VUs까지 부하를 걸었을 때 결제 단계부터 실패율이 급증하는 패턴을 확인([`../k6/README.md`](../k6/README.md) 실행 결과 참고).
- **온프레미스 한계 지점을 실제로 관측**: 동시접속 VU를 0→2000까지 램프업하자 노드 수는 4개로 고정된 채 Running 파드가 33개까지 늘었고, 이후 **Pending 파드가 0→7개까지 쌓이면서 P95 레이턴시가 6.54초, k6 실패율이 최대 6%까지** 치솟는 것을 확인했습니다. 고정 자원이 한계에 도달하면 무슨 일이 일어나는지를 보여주는 이 실험의 핵심 증거입니다.

  ![온프레미스 한계 도달 — Pending 파드 발생](images/limit-found-pending-pods.png)

- **온프레미스 vs EKS 비교 완료**: Grafana에 양쪽 Prometheus(`prometheus-onprem`/`eks`)를 동시에 붙인 반반 비교 대시보드로 동일 부하(최대 300 VU)를 걸어 직접 비교했습니다.
  - 부하 투입 전(베이스라인)에는 온프레 4노드/EKS 8노드 모두 Pending 0으로 안정적이었습니다.
  - 부하가 유지되자 온프레는 노드가 4개로 고정된 채 **Pending이 6\~7개까지 쌓여 수 분간 유지**됐고, EKS는 Karpenter가 노드를 3→8개로 늘렸지만 **Pending도 최대 12개까지 두 차례 스파이크친 뒤에야 해소**됐습니다 — Karpenter가 있어도 노드 프로비저닝(60\~90초)이 끝날 때까지는 온프레와 마찬가지로 Pending이 쌓인다는 걸 확인했습니다.
  - 300 VU 부하 테스트 전 과정(k6 결과 패널 + 인프라 메트릭 패널)은 영상으로 남겼습니다: [`videos/onprem-vs-eks-300vu-load-test.mp4`](videos/onprem-vs-eks-300vu-load-test.mp4)

## 회고

- **"다 한 것 같은데" 함정**: AI 가이드를 따라가면 끝난 것처럼 보여도 설치 항목을 하나씩 빠뜨려 재설치를 반복하게 됨 — 결국 절차를 직접 문서로 고정하는 것이 재현성의 핵심이었습니다.
- **측정 환경은 설계의 일부**: 운영 파드가 실험 워커에 섞이면 "고정 자원의 순수 한계"를 측정할 수 없어, worker3를 taint로 격리해 실험 워커를 순수 부하 전용으로 만들었습니다.
- **실험은 조건이 90%**: 부하 세기뿐 아니라 유지 시간, 재고 사전 세팅 등을 맞추지 않으면 결과가 오염됩니다.
